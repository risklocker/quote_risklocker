# Risklocker Project Memory

## Current Snapshot

- Risklocker is a private internal motor-quotation converter. Its staff workflow is Upload -> Check Values -> Generate PDF.
- The repository uses a FastAPI backend, Next.js frontend, Supabase/Postgres for application data, and private Supabase Storage for source and generated PDFs.
- The only persistent PDF provider is private Supabase Storage. SharePoint/OneDrive is an optional backend-only archive and is not a runtime requirement.
- PDF output is deterministic HTML/CSS rendering from saved template configuration and reviewed draft values. AI must not generate final PDF layouts.
- Extraction may auto-fill clear values, but ambiguous, conflicting, or missing required values must remain `Check Needed` for staff review.

## Current Architecture Facts

- Backend entry points are `backend/app/main.py` and `backend/app/api/routes.py`; frontend routes are under `frontend/src/app/`.
- Source and generated PDF binaries have rolling 30-day storage retention. Their metadata, extraction records, reviewed values, templates, and version snapshots remain after binary expiry.
- The default motor template is locked and copy-only. Editable templates are configured in the admin A4 builder.
- Backend roles are Staff, Manager, and Admin. Authorization belongs in backend routes and services, never only in frontend UI.
- Authentication is passwordless for named `@risklocker.com` employee accounts. SMTP-delivered one-time codes create revocable Postgres sessions exposed only through secure HttpOnly cookies; the browser stores no authentication token.
- New employee accounts begin in an `invited` state and receive an invitation email containing a one-time login code. The account auto-promotes from `invited` to `active` on the first successful code verification.
- In-app notifications record invitations, role changes, and account-status changes per recipient. The application shell displays an unread count and an Inbox page provides mark-read actions.
- The backend SMTP relay delivers invitation emails, role-change notices, status-change notices, and an Admin-only test message in addition to on-demand login codes.

## Verified Technical Baseline (2026-07-17)

- The full automated check is green: 90 backend tests pass and the frontend production build succeeds.
- Notifications, invitations, role-change emails, and the in-app inbox are implemented and tested. SMTP delivery supports multiple message types: invitation with login code, role-change notice, status-change notice, and Admin self-test.
- New employee accounts are created in `invited` status and auto-promote to `active` on first successful code verification.
- **High, open:** the upload screen advertises image formats and 50 MB files, while backend validation accepts PDFs only and the configured default limit is 1 MB.
- **Medium, open:** `ENHANCED_READING_ENABLED`, `STRICT_NO_GUESSING`, and `AUTO_DOWNLOAD_GENERATED_PDF` are loaded as settings but are not used outside configuration loading.
- **Medium, open:** mutable JSON API endpoints outside authentication and user management use untyped dictionaries; non-authentication route/RBAC integration tests, browser workflow tests, and repository CI are still missing.
- **Medium, product/UX decision:** dictionary and extraction-settings APIs have no dedicated screens, and the shared shell does not filter its Admin link by client-side role. Backend authorization remains authoritative.
- **Operational verification required:** repository code does not configure in-app rate limiting or security headers; any platform or gateway controls are currently undocumented and unverified.
- Microsoft 365 archive work remains intentionally optional, not a current defect.
